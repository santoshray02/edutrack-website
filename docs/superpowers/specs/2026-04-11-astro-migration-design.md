# Astro Migration Design — edunodex.in

**Date:** 2026-04-11
**Status:** Approved for autonomous execution (user delegated decisions)
**Scope:** Option B from brainstorm — platform migration + targeted fixes, no redesign

## Goals

1. **Fix mobile color rendering.** Add `color-scheme: light` and switch Tailwind `dark:` variants from auto (`media`) to opt-in (`class`), so phones with system dark mode render the site correctly.
2. **Eliminate cross-file duplication.** Nav, footer, analytics, SEO tags, structured data — currently copy-pasted across 20+ HTML files. After migration, one edit propagates everywhere.
3. **Move off Tailwind v3 CDN** (development-only, ships full engine at runtime) to **Tailwind v4 build** via first-party Vite plugin — CSS+JS under 25KB gzipped.
4. **Blog authoring in MDX** — convert 14 HTML articles to Markdown + frontmatter so content edits don't require touching HTML.
5. **Zero visible change for end users.** Same URLs, same content, same design, same SEO signals. Only observable difference: faster loads and correct mobile colors.
6. **Preserve SEO:** all URLs identical, all canonical tags, OG tags, Twitter cards, JSON-LD structured data (SoftwareApplication, Organization, FAQPage, WebSite), sitemap.xml, robots.txt, CNAME.

## Constraints

- **Live production site** at `edunodex.in`. CNAME file is at repo root. Currently served by GitHub Pages from `main` branch root (no CI, raw HTML).
- **`gh` CLI is not authenticated** — cannot change GitHub Pages source via API. ⇒ Built HTML output MUST live at repo root so Pages config stays untouched.
- **Node 18.19.1 + Bun 1.3.12** available locally.
- **User is away during execution** — zero margin for breaking live site. Every step verified locally before push.

## Key design decisions

### 1. Source in `astro/` subfolder, built output at root

Chosen because: Pages currently serves from main root, and I can't change Pages settings without `gh` auth. Source files (`src/`, `package.json`, `astro.config.mjs`) don't conflict with root HTML because they're under `astro/`. `bun run build` outputs to `astro/dist/`, which gets rsynced to repo root (overwriting old HTML, preserving `astro/`, `.github/`, `.git/`, docs).

### 2. Force light mode

- Every page gets `<meta name="color-scheme" content="light">` in head (in `BaseLayout`).
- Tailwind config uses `darkMode: 'class'` — so `dark:` variants only apply when a `dark` class is present on `<html>`, which it never will be.
- No `dark:` classes removed from existing pages; they simply become dormant. Future dark mode = add a toggle that sets `dark` class.

### 3. Blog content collection

`src/content/config.ts` defines a `blog` collection with Zod schema (title, description, publishedDate, updatedDate, author, category, readTime, heroImage, keywords, canonical).

14 articles are manually converted from HTML → MDX, preserving all content, images, links, and SEO metadata. Each article's frontmatter drives the article layout's `<SEO>` and `<StructuredData>` slots.

Dynamic route `[...slug].astro` picks up all collection entries. URLs preserved exactly: `/blog/<slug>/`.

### 4. Analytics and tracking

Preserved verbatim from existing pages:
- `gtag` IDs: `GT-MKBHFCK6`, `G-9B2DLLXH62`, `AW-18055680778`
- All `gtag('event', 'click', ...)` tracking on CTAs
- Centralized in `Analytics.astro` component, injected once in BaseLayout.

### 5. Structured data

Page-specific JSON-LD blocks (SoftwareApplication, Organization, FAQPage, WebSite on homepage; BlogPosting on articles; WebPage on others) live in page files and get rendered via a `<StructuredData>` slot in BaseLayout.

### 6. Tailwind v4 setup

Astro uses the official `@tailwindcss/vite` plugin. Config lives in `tailwind.config.mjs` (for `darkMode: 'class'` and content paths). `src/styles/global.css` contains `@import "tailwindcss";` and any custom layers.

### 7. GitHub Actions workflow

File: `.github/workflows/deploy.yml`

Trigger: `push` to `main`, filtered by `paths: ['astro/**', '.github/workflows/deploy.yml']` — avoids infinite loop with commits the workflow itself pushes (which only touch root HTML, not `astro/**`).

Steps:
1. Checkout
2. Install Bun (`oven-sh/setup-bun@v2`)
3. `cd astro && bun install --frozen-lockfile && bun run build`
4. `rsync -a --delete-excluded --exclude='astro/' --exclude='.github/' --exclude='.git/' --exclude='docs/' --exclude='.superpowers/' --exclude='README.md' --exclude='DEPLOY_NOW.md' --exclude='GENAI_2026_UPDATE.md' --exclude='GITHUB_PAGES_DEPLOYMENT.md' --exclude='WHATSAPP_AUTO_REPLY_SETUP.md' astro/dist/ ./`
5. `git add -A && git diff --cached --quiet || git commit -m "build: deploy Astro output [skip ci]" && git push`

Requires `permissions: contents: write`.

For the initial migration, build happens **locally** (via Bun) and is committed alongside the source in one cutover commit. The workflow is for future rebuilds after content edits.

## Rollback plan

If anything goes wrong after merge to main:

```bash
git revert <merge-commit-sha>
git push origin main
```

Pages rebuilds in ~30s, old HTML is restored byte-for-byte.

## Out of scope

- Dark mode toggle (disabled sitewide for now)
- Redesign / new components (Option C)
- Changing blog URLs
- Migrating `ads/` to Astro components (preserved as static files in public/ads/)
- Changing GitHub Pages source setting (would need user intervention)
